Introduction

There are a lot of introductory articles on the internet. Although these write-ups are useful to get you started, it usually leaves you in a state of uncertainty. Take for example the Model-View-ViewModel pattern, since it started gaining momentum a couple of years ago there has been hundreds of "Introduction to MVVM" articles popping up around the web, still to this day. They all claim to be different, but it usually boils down to something like this: "The views are the visual elements like windows and data templates, the view models are abstractions of the views and encapsulates the presentation logic, while the model is the data and the business logic of your application". This can leave you with all sorts of unknowns. If the model is the data, we can define POCO classes and use an ORM to map them to a database, right? Sure, this will be fine for a single-user application where the database lives on the same computer, but what if you want to hide the data access and business logic behind a service layer? The MVVM pattern is a front-end architectural pattern, does this mean that we are dealing with _VVM when we move the M to the back-end? It was quite evident from reading the comments of those articles that a lot of less experienced developers were left confused and frustrated.

Look at it this way, my article and example code will give you an overview of the architecture for the entire end-to-end application. I wanted to present a production-ready real world example, following all the best practises and patterns. I've tried to implement commonly used technologies, but obviously I can't go into detail about every technology, but that's what the introductory articles are for!

Background

I have developed the entire system from scratch using Visual Studio 2015 Community Edition, based on the .NET Framework version 4.5.2. For data storage and queries I use a SQL Server 2014 database. The following technologies were used to develop the solution:

  • Windows Presentation Foundation (WPF) - The front-end application.
  • Windows Communication Foundation (WCF) - Enables communication between processes or tiers.
  • Unity Application Block - Dependency Injection Container that facilitates loose coupling of components, and coding against contracts instead of concrete implementations.
  • Entity Framework - The Object Relational Mapper (ORM) that maps our Models to database objects, and takes care of the data storage implementation details.
  • Log4Net - A very sophisticated logging library.
  • Visual Studio Unit Testing - The standard Unit-testing framework that ships with Visual Studio.
  • UI Automation Framework - The Microsoft-provided framework for automation that we use to write scripted UI test scenario's.
  • Moq - The powerful mocking library used in testing to mock dependencies.

The following patterns and principles can be spotted throughout the solution:

  • SOLID Principles - Separation of Concerns, Open-Closed principle, Liskov Substitution, Interface Segregation, and Dependency Inversion.
  • Model-View-ViewModel - Front-end architectural pattern.
  • Repository Pattern - Decoupling the data storage infrastructure from the business logic, making the code Data Storage agnostic.
  • Factory Method
  • Layered Architecture - The code will be broken up into layers in accordance with the separation of concerns principle.
  • Service Oriented Architecture - The front-end application will communicate with a couple of microservices to access the Application Programming Interface of the system.
